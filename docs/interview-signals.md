> 采集:2026-07-11 · 来源为求职博主真实面经(细节具体,可信度较高)
> ⚠️ 该博主同时卖课/卖项目,其项目设定(MindBridge / 6类agent / 风控)勿照抄,只提取**工程考点**
> 用途:反推 AgentFlow 应覆盖哪些能力点,每个考点 = 一个简历可讲的特性

## 一、腾讯 AI 应用「一面」考点(围绕候选人的 agent 项目拷打)

1. 项目完整链路 + 技术栈
2. Harness 在项目里的作用、为什么这么设计
3. 多 agent 协作具体怎么协作(他有 6 类 agent)
4. 风险判断:基于规则还是模型?
5. **上下文注入怎么做,如何避免过多历史消息进模型**
6. AgentContext 的作用
7. **Run Trace(执行轨迹)怎么保存,保存时机是什么**
8. 高风险预警闭环:**怎么证明闭环真的执行成功**
9. Skills 怎么动态加载
10. **怎么保证 Skill 产出内容稳定,而不是模型自由发挥**
11. 手撕:动态异或图最大路径值(hard 变体)
12. 🔑 反问透露信号:面试官关注「**更看重 Agent 的 autonomous capability,还是可控可运营的 workflow agent**」

## 二、腾讯 AI 应用「二面」考点(深挖分布式系统工程 —— 与 AgentFlow 高度重合)

1. Skills 调优心得
2. 为什么把 Excel 写入/预警发送**封成 MCP 工具**,而不直接调 service 方法(抽象理由)
3. **回复生成成功、但落库和工具任务失败,这一轮 turn 的最终状态如何定义**(部分失败语义)
4. **Harness 与 Agent Runtime 的边界怎么定义**
5. 高风险内容执行到一半失败,**Harness 应保存哪些数据**(容错持久化)
6. **MCP 路径和队列路径都能触发副作用,怎么保证所有副作用路径经过同一套 policy**(一致性)
7. **高风险 alert 失败进入 dead letter 后,谁来恢复**(死信恢复)
8. 多个 skill 同时命中怎么排序
9. **工具队列如何保证幂等**
10. 安全处理是代码层 enforce 还是靠提示词
11. **同一高风险 report 重复入队,会不会重复发邮件/建 case/写 Excel**(去重/幂等)
12. 手撕:最大子数组(easy)

## 三、阿里 AI agent 研发「二面」(现场 vibe coding + 工程判断)

- 现场用 AI 编码,考的是工程判断力:
  - 任务怎么拆,有没有更细粒度的拆法
  - 为什么要写测试
  - 「任务完成」怎么定义(能跑就算完成吗)
  - 还有哪些可优化 / AI 写得不好的地方
  - AI 写的是否过度复杂
  - 边界条件怎么定义
  - 不用 vibe coding 你自己多久能做完
- 启示:**代码要真自己写、能讲清每个决策、能批判性看待实现(拆分粒度/测试/definition of done/边界/过度设计)**

## 四、对 AgentFlow 的启示(考点 → 特性映射)

| 面试考点 | AgentFlow 是否已覆盖 | 待补强 |
|---|---|---|
| 幂等(工具队列/重复入队去重) | ✅ 已有(Redisson+MD5) | 明确"副作用级幂等",防重复发通知/写库 |
| 死信队列 + 谁来恢复 | ✅ 已有死信 | 补"死信恢复流程/重放机制" |
| 部分失败的 turn 状态语义 | 🔶 状态机有,但未细化 | **新增:一次任务多子步骤部分成功/失败的最终态定义** |
| 副作用统一经过一套 policy | 🔶 有 LLM 网关 | **新增:统一副作用出口/policy 层(所有写库+通知过同一闸)** |
| Run Trace 执行轨迹 + 保存时机 | 🔶 只写了"基础可观测" | **升级为 Run Trace:每步/工具调用/状态变更留痕,可回放** |
| Harness vs Runtime 边界 | 🔶 隐含 | **架构图明确标注边界:编排/网关(harness) vs worker内agent执行(runtime)** |
| 上下文注入 / 防上下文膨胀 | ❌ 未涉及 | **新增:AgentContext + 上下文裁剪/摘要,控制进模型的历史量** |
| Skill 动态加载 + 输出稳定 | 🔶 未强调 | **新增:工具/技能动态注册 + schema 约束输出(不让模型自由发挥)** |
| 多 skill 命中排序 | ❌ | 可选:工具选择的排序策略 |

## 五、定位微调(重要)

腾讯反问明确透露:大厂更关注「**可控、可运营的 workflow agent**」而非纯 autonomous。
→ AgentFlow 的 infra/可控定位 **100% 命中**。叙事关键词锁定:
**可控(controllable)、可运营(operable)、可观测(observable/traceable)、可恢复(recoverable)、幂等(idempotent)**。

## 六、诚实提醒

- 这些考点是**别人项目**被问的,不代表你的项目一定被这么问;但它们精准反映了大厂 AI 应用岗对**系统工程能力**的偏好,可当 AgentFlow 的"验收清单"。
- 博主项目(MindBridge/风控/6类agent)**只借鉴工程设计点,勿抄设定**(撞脸风险)。
- 手撕仍是硬门槛(一面 hard、二面 easy)→ **LeetCode 不能停**。
